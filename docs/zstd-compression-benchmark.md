# ZSTD Compression Level Benchmark

Benchmark of Parquet ZSTD compression levels on both synthetic and **real
production data** from E2E Docker compose (OTEL-style structured logs and traces
with promoted columns, MAP columns, bloom filters on `service.name` and
`trace_id`). Measured on Apple M-series (18 cores).

## Configuration Mapping

Victoria Lakehouse maps `--lakehouse.insert.compression-level` (1-22) to
`parquet-go/compress/zstd` levels:

| Config Level | zstd.Level              | Description |
|-------------|-------------------------|-------------|
| 1           | SpeedFastest            | Minimal compression, maximum throughput |
| 2-5         | SpeedDefault            | Balanced (parquet-go default) |
| 6-10        | SpeedBetterCompression  | Better ratio, moderate CPU cost |
| 11-22       | SpeedBestCompression    | Best ratio, high CPU + memory cost |

Current default: **level 7** (`SpeedBetterCompression`). Changed from level 3
based on real-data benchmarks showing 25% smaller logs and 16% smaller traces
with only 15-20% slower writes and negligible read impact.

## Key Findings

### Real Data vs Synthetic: Compression ratios differ significantly

| Signal | Level 3 Ratio (Real) | Level 3 Ratio (Synthetic) | Difference |
|--------|---------------------|--------------------------|------------|
| Logs   | 4.60x               | 7.55x                   | Real data compresses 39% less |
| Traces | 7.93x               | N/A                      | Traces compress much better than logs |

Real E2E data has higher entropy than synthetic data — unique trace IDs, varied
log bodies, diverse attribute values. Synthetic benchmarks overestimate
compression ratios by ~40% for logs. **Always benchmark with real data.**

### Compression Ratio (Real Data)
- **Level 1 → 3**: Logs +3.7%, Traces +14.8% better — good tradeoff
- **Level 3 → 7**: Logs **+25% better**, Traces +15.6% better — **excellent tradeoff**
- **Level 7 → 11**: Logs +1.7%, Traces +2.9% better — **poor tradeoff**
- Levels 3 and 5 produce **identical output** (parquet-go maps both to SpeedDefault)
- Levels 7 and 9 produce **identical output** (both map to SpeedBetterCompression)

### Write Performance (Real Data)
- Level 1: 341-393 MB/s — easily saturates 1 Gbps ingest
- Level 3: 322-370 MB/s — 94% of fastest, still saturates 1 Gbps
- Level 7: 257-335 MB/s — 80-91% of level 3, moderate cost
- Level 11: 63-95 MB/s — **5x slower than level 1**, problematic for high-volume ingest

### Read Performance (Real Data) — Read latency is nearly flat
- Read speed varies only 1.3x across all levels (vs 5.4x for write)
- Level 7 reads are 74-85% of level 3 speed — acceptable for query workloads
- Level 11 reads are 72-81% of level 3 — marginal degradation
- **Decompression is NOT the bottleneck** — row materialization dominates read time

### Memory (Real Data)
- Logs at scale (378K rows): 164-198 MB across all levels — <20% difference
- Traces: 53-86 MB — level 7 actually uses less memory than level 3
- Level 11 uses ~20% more write memory, not the 2-3x seen in synthetic tests

## Recommendation

### Default: Level 7 (current) — Best cost/performance compromise

Level 7 (`SpeedBetterCompression`) is the optimal default based on real-data benchmarks:
- **6.13x logs, 9.39x traces** compression on real production data
- **25% smaller logs, 16% smaller traces** vs level 3
- Write speed 257-335 MB/s — still saturates most ingest pipelines
- Read speed only 15-26% slower than level 3 — decompression is not the bottleneck
- Memory within 5% of level 3 for logs, actually less for traces
- Saves **$50.70/month per 2 TB/day** vs level 3 ($608/year)

### When to change

| Scenario | Recommended Level | Why |
|----------|------------------|-----|
| High-volume ingest (>500 MB/s) | 1 | Maximize throughput, 4.4x logs / 6.9x traces |
| Maximum write speed, acceptable ratio | 3 | 322+ MB/s writes, 4.6x logs / 7.9x traces |
| General purpose (default) | **7** | Best cost/performance, 25% smaller than level 3 |
| Archival / write-once data | **7** | Best ratio-per-CPU tradeoff |
| Never use | 11+ | <2% better ratio costs 5x CPU, read gain is zero |

Level 11+ is never recommended: the jump from SpeedBetterCompression to
SpeedBestCompression buys only 1.7% better ratio at 5x the write CPU cost.
Read performance is unchanged since decompression is not the bottleneck.

## Appendix: parquet-go/compress/zstd Level Mapping

The `parquet-go` library uses `klauspost/compress/zstd` internally. The four
speed levels map to klauspost encoder options:

| zstd.Level              | klauspost Option               | Window Size |
|-------------------------|-------------------------------|-------------|
| SpeedFastest            | WithEncoderLevel(SpeedFastest) | 64 KB      |
| SpeedDefault            | WithEncoderLevel(SpeedDefault) | 128 KB     |
| SpeedBetterCompression  | WithEncoderLevel(SpeedBetter)  | 256 KB     |
| SpeedBestCompression    | WithEncoderLevel(SpeedBest)    | 512 KB     |

Only 4 effective levels exist despite the 1-22 config range. Levels within the
same bucket produce identical output.
